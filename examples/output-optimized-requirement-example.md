# Optimized Requirement Example

## Requirement Summary
商家可对符合售后条件的订单发起退款申请。财务负责审批退款申请。审批驳回时，系统必须记录驳回原因并通知商家。审批通过时，系统负责执行原路退款并根据结果更新退款申请和订单状态。若原路退款失败，人工处理人员负责介入并给出最终处理结果，直到退款申请进入明确终态。

## Storyline
1. 商家提交退款申请。
2. 财务审核退款申请。
3. 若审核驳回，系统记录驳回原因，通知商家，退款申请状态变为已驳回，流程结束。
4. 若审核通过，系统执行原路退款。
5. 若退款成功，退款申请状态变为已完成，订单状态变为已退款，商家与相关业务方可查看结果，流程结束。
6. 若退款失败，系统创建人工处理任务并指派给人工处理人员。
7. 人工处理人员负责跟进退款失败原因，完成补偿处理或确认关闭。
8. 人工处理完成后，必须回写最终结果：
   - 若退款完成，退款申请状态变为已完成，订单状态变为已退款。
   - 若退款无法完成，退款申请状态变为已关闭，并记录关闭原因。
9. 最终结果必须对商家和相关业务方可见。

## Key Responsibilities
- 商家：发起退款申请
- 财务：审批退款申请并给出驳回原因
- 系统：执行原路退款、更新状态、通知结果、创建人工处理任务
- 人工处理人员：处理退款失败场景并确认最终结果

## Core Rules
- 每个退款申请必须有唯一明确的责任流转。
- 每条分支都必须进入可观察的最终状态。
- 驳回和关闭都必须记录原因。
- 人工介入后必须由人工处理人员确认最终结果，不允许无限悬置。

## Final States
- 已驳回
- 已完成
- 已关闭
